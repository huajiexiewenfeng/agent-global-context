# Agent Global Context 设计

## 目的

Agent Global Context 是一个面向 AI Agent 的 Markdown-first 全局上下文层。它用于跨 session 保存会影响未来 agent 行为的关键信息，包括用户背景、技术偏好、代码习惯、工作环境和项目信息。

它不是完整聊天记录系统，而是一个选择性、可审计、可维护的长期上下文系统。

Memory 文件只是存储机制。真正的产品目标是 agent global context：让 agent 获得可持续的全局理解，从而在未来做出更好的协作和工程决策。

## 非目标

- 默认不保存完整聊天日志。
- 不替代模型当前上下文窗口。
- MVP 不依赖数据库、向量库或后台服务。
- 不把未经确认的个人判断、心理推断或人格推断当作长期事实。

## 原则

1. 记录关键上下文，而不是记录全部内容。
2. 记忆内容使用 Markdown，行为配置使用 YAML。
3. 根据优先级和任务相关性读取上下文。
4. 长期记忆必须标注来源和置信度。
5. 敏感或高影响个人记忆需要用户确认。
6. `P0-P2` 长期记忆必须说明为什么重要；`P3-P4` 在有帮助时建议写 rationale。

## 优先级模型

- `P0`：用户背景、稳定身份级上下文、语言偏好、长期协作需求。
- `P1`：技术偏好、工具偏好、架构偏好、工作流偏好。
- `P2`：代码风格、测试习惯、review 习惯、分支和提交约定。
- `P3`：项目知识、环境知识、命令、依赖、架构决策。
- `P4`：当前任务状态、session 摘要、下一步、阻塞点。

## 读取模型

读取从 `index.md` 开始，然后按优先级读取相关上下文：

- `P0`：默认读取短 `Recall Summary`。
- `P1`：讨论技术选择、工具、架构或协作流程时读取。
- `P2`：写代码、review、测试、重构时读取。
- `P3`：当前工作目录或用户请求命中某个项目时读取。
- `P4`：用户要求继续上次工作或恢复任务状态时读取。

## 写入模型

写入分为四个概念阶段：

- `observe`：发现可能有长期价值的信息。
- `stage`：把候选信息放入暂存区。
- `commit`：把已确认或高置信度的信息写入长期记忆。
- `reject`：明确不保存，或标记为被拒绝。

MVP 实现 `recall`、`commit`、alpha 自动捕获和候选审查。

自动捕获默认开启。它只把候选写入 staging，绝不直接产生长期记忆。用户可以在配置中关闭。

模板中包含 staging 文件，但 MVP agent 只有在用户明确要求暂存记忆，或敏感/高影响候选信息不适合直接提交时，才应使用 staging。

## 自动捕获

自动捕获是 alpha 阶段的 `observe` 实现。

开启后，agent 可以识别可能有长期价值的信号，并把候选条目写入 `staging/inbox.md`。自动捕获只产生候选，绝不直接产生长期记忆。

只捕获强信号：

- 用户偏好或习惯
- 长期项目或工作流决策
- 用户对 agent 行为的纠正
- 稳定身份级上下文，例如背景、角色、语言或工作环境

明确不捕获：

- 一次性任务细节
- 临时状态
- 尚未决定的探索性想法
- 闲聊或无关讨论
- 未确认的心理、人格、动机或情绪状态推断

候选条目应包含 source、proposed priority、proposed confidence、rationale、created time 和 status。

敏感过滤必须在 observe 阶段生效。密钥和凭据不得进入 staging。个人敏感上下文只有在可能确实值得保存且询问用户时，才进入 pending review。未确认的心理或人格推断默认丢弃。

`staging/inbox.md` 不进入默认 recall。候选不是长期事实。

操作原则：

```text
自动捕获可以发现。
暂存区可以承载。
审查可以批准。
提交才能成为长期记忆。
读取时不能把未确认候选当事实。
```

## 候选审查与提升

候选通过 review 和 commit 进入正式记忆。

```text
auto_capture
  -> staging/inbox.md 或 staging/pending-review.md
  -> review
  -> commit
  -> long-term memory
```

Review 结果：

- `promoted`：接受并提交到长期记忆。
- `rejected`：被用户或策略拒绝。
- `expired`：超过保留期未审查。
- `pending`：仍等待审查。

提升规则：

- `confirmed`：可以提升。
- `observed`：非敏感且来源可靠时可以提升。
- `inferred`：需要用户确认。
- `tentative`：需要确认或重复证据。

候选被提升时，staging 条目应记录 `promoted_to`、`reviewed_at` 和 `status: promoted`。

当候选数量或年龄超过配置阈值时，agent 应提示 review。例如待审候选不少于 5 条，或最旧待审候选已超过 7 天。Review 提示不应阻塞正常工作，除非涉及敏感/高影响确认或过期处理。

## 读取预算

优先级决定重要性，相关性决定是否读取，预算决定读取深度。当多个优先级同时命中时，agent 应先读取摘要，只在必要时读取详细条目。

默认顺序：

1. `index.md`
2. `P0` `Recall Summary`
3. 命中项目的 `P3` summary
4. 相关 `P1/P2` summary
5. 仅在继续任务时读取 `P4` active state

上下文紧张时，应先丢弃旧 session 细节，而不是丢弃活跃项目上下文或直接相关偏好。

## 置信度

使用四个 confidence 值：

- `confirmed`：用户明确陈述、批准，或要求记住。
- `observed`：来自可靠文件、命令输出、配置，或重复可见行为。
- `inferred`：agent 根据上下文推导，但没有直接确认。
- `tentative`：证据弱、有歧义，或可能只是临时状态。

## 项目和去重规则

项目级记忆应通过稳定 project id 管理。识别顺序是：`index.md` 的 `Project Map`、Git remote slug、当前目录 slug。新的或有歧义的 project id 需要用户确认。

用户确认过的项目绑定是权威来源，应优先于 Git remote slug 或目录名这类启发式规则。

追加 memory item 前，agent 应先扫描目标文件，检查是否已有语义相同的记忆。能更新已有记忆时优先更新；当新信息替代旧信息时，使用 `status: superseded` 标记旧条目。

## 冲突和生命周期规则

当前用户指令优先于已存记忆，除非该指令不安全或不可执行。在活跃项目中，已确认的项目约定和项目决策优先于通用个人偏好。

`P4` 是临时状态。active state 应原地更新，已完成的下一步应移除，过期阻塞点不应被带入无关任务。

## 敏感记忆

敏感个人信息、密钥、凭据、私钥、财务信息、医疗信息，或未经确认的心理/人格推断，不能被静默写入长期记忆。Agent 应询问用户是写入、暂存待审，还是跳过。
