# Thinking Skills 聊天运行时设计

## 目的

构建一个轻量级的服务端聊天运行时，让微信小程序可以使用现有的 thinking skills，为亲近的家人提供安全的情感支持聊天机器人。

这个产品不是 Codex 的服务端复制版。它是一个范围更窄的聊天服务：加载选定的 skill 指令，接收微信小程序传来的对话上下文，在服务端调用可配置的模型提供方，并把有情感支持感的回复返回给微信小程序。

## 目标

- 为家人用户提供微信小程序聊天体验。
- 模型提供方 API key 和微信相关密钥只保存在服务端。
- 复用现有 `thinking-router`、`emotional-support`、`safety-boundary` 的 skill 行为。
- 支持简短、温暖、低术语感的情感支持对话。
- 识别危机信号，并切换到安全优先的回应路径。
- 默认由手机本地保存 session 状态和最近聊天上下文。
- 避免在服务端存储完整的敏感聊天记录。

## 非目标

- 不构建完整兼容 Codex 的 agent runtime。
- 不向用户暴露 shell、文件系统、代码编辑或开发者工具能力。
- 不实现诊断、治疗、医疗建议或专业心理健康服务。
- 第一版不做支付、后台管理系统或公开用户注册流程。
- 第一版不支持所有 thinking skills。
- 不存储长期心理画像。

## 推荐方案

使用 Node.js、TypeScript 和 Fastify 实现一个轻量后端。后端对微信小程序暴露一个主要聊天接口，并在服务端调用可配置的模型提供方。

第一版不要求服务端保存对话内容。如果需要服务端存储，SQLite 可用于 session 元数据、轻量用量记录或未来的摘要能力。数据结构应保持小而可替换，方便之后迁移到 PostgreSQL，同时不影响微信小程序 API。

## 系统架构

```text
微信小程序
  -> 后端聊天 API
    -> 请求鉴权
    -> Session 处理
    -> Skill 运行时
    -> 安全层
    -> 模型提供方客户端
    -> 回复后处理
  -> 微信小程序
```

## 主要请求流程

```text
1. 微信小程序向 POST /chat 发送用户消息。
2. 后端验证请求，并解析用户和 session。
3. 后端加载选定的 thinking skill 指令。
4. 安全层检查是否存在即时风险信号。
5. 如果检测到风险，后端使用安全回应路径。
6. 如果没有检测到风险，后端使用 emotional-support 指令。
7. 后端结合最近上下文或摘要组装模型输入。
8. 后端调用配置好的模型提供方。
9. 后端对回复做语气和安全后处理。
10. 后端把 assistant 消息返回给微信小程序。
11. 后端可选更新轻量 session 摘要。
```

## API 设计

### `POST /chat`

请求：

```json
{
  "userId": "family-user-id",
  "sessionId": "local-session-id",
  "message": "用户消息",
  "recentMessages": [
    {
      "role": "user",
      "content": "上一条用户消息"
    },
    {
      "role": "assistant",
      "content": "上一条助手消息"
    }
  ]
}
```

响应：

```json
{
  "sessionId": "local-session-id",
  "message": {
    "role": "assistant",
    "content": "助手回复"
  },
  "safety": {
    "level": "normal",
    "reason": null
  }
}
```

响应从一开始就应该使用稳定的外层结构。这样微信小程序之后可以增加安全提示条、重试行为或人工支持建议，而不需要重新设计 API。

## Session 策略

微信小程序是默认的 session 持有方。它可以缓存：

- `userId`
- `sessionId`
- 最近可见的聊天消息
- 本地 UI 状态
- 用户偏好设置

微信小程序不能缓存：

- 模型提供方 API key
- 微信 AppSecret
- 后端签名密钥
- 长期敏感标签
- 隐藏的安全分类结果

后端默认不保存对话内容。它可以存储：

- session 元数据
- 简短对话摘要，但只能在明确启用时保存
- 时间戳
- 模型用量元数据

后端默认不能存储完整的敏感聊天记录。如果之后确实需要完整历史记录，必须把它作为明确的产品决策，并配套清楚的用户隐私说明。

## Skill 运行时

第一版支持固定的 skill 集合：

- `thinking-router`
- `emotional-support`
- `safety-boundary`

运行时应把 skill 文件视为服务端拥有的 prompt 资产。它可以在启动时或按需加载这些文件，把它们规范化为模型指令，并让应用逻辑和 skill 文本保持分离。

第一版可以把大多数普通消息直接路由到 `emotional-support`。只有当消息比较模糊，或以后增加更多 domain 时，再把 `thinking-router` 作为轻量分类步骤使用。

## 安全行为

安全检查发生在普通情感支持生成之前。

如果用户消息暗示自伤、伤害他人、虐待、胁迫、严重功能受损、医疗紧急情况或即时危险，后端必须切换到安全优先回应。这个回应应该：

- 避免深度分析
- 避免诊断
- 避免淡化情况
- 鼓励用户寻求现实世界里的即时支持
- 在合适时建议联系可信任的人或当地紧急服务
- 保持语气冷静、直接、简短

产品不能把自己呈现为治疗、诊断、医疗服务或专业支持的替代品。

## 模型提供方集成

后端使用服务端 API key 调用可配置的模型提供方。API key 存储在环境变量中。

第一版应优先支持 OpenAI-compatible chat API，因为很多模型服务和中转服务都兼容这种接口形态。应用必须把 provider 放在 adapter 后面，让 thinking skills runtime 不依赖某一个固定模型厂商。

可支持的 provider 包括国内模型服务、自控的海外服务，或用于个人测试的临时中转服务。中转服务不应被视为敏感家庭对话的长期默认方案，除非它的隐私、稳定性和合规边界已经被确认。

必需环境变量：

```text
MODEL_PROVIDER
MODEL_BASE_URL
MODEL_API_KEY
MODEL_NAME
```

可选环境变量：

```text
DATABASE_URL
SESSION_SECRET
WECHAT_APP_ID
WECHAT_APP_SECRET
```

第一版应该让模型提供方、base URL 和模型名称可配置，而不是在整个代码库里硬编码某个厂商。

## 错误处理

后端应该返回稳定的错误结构。

预期错误类别：

- 请求体无效
- 请求未授权
- session 缺失
- 模型提供方超时
- 模型提供方错误
- 安全回应兜底
- 服务端内部错误

当模型调用失败时，微信小程序应该收到温和、可重试的提示，而不是原始 provider 错误。

## 隐私和安全

- 所有 provider 密钥都保存在后端。
- 不把模型提供方 API key、微信 AppSecret 或后端密钥发送给微信小程序。
- 生产环境使用 HTTPS。
- 在微信小程序后台把后端域名配置为合法请求域名。
- 生产环境默认避免记录原始用户消息。
- 从错误日志中移除敏感字段。
- 在分享给初始家人用户之外的人之前，增加限流。

## 测试策略

单元测试：

- 请求 schema 校验
- skill 加载
- prompt 组装
- 安全分类包装器
- 错误响应格式化

集成测试：

- `POST /chat` 普通情感支持消息
- `POST /chat` 类危机消息使用安全路径
- 模型超时返回稳定兜底
- 缺少 API key 时在启动或健康检查阶段失败

手动验证：

- 从微信小程序或 API 客户端发送一段普通家人支持对话
- 确认前端代码或网络响应里没有 API key
- 确认敏感 provider 错误不会展示给用户
- 确认类安全风险消息不会收到普通 coaching 回复

## 第一个里程碑

第一个里程碑是一个本地后端 MVP，包含：

- Fastify server
- `POST /chat`
- 基于环境变量的模型提供方配置
- 固定 emotional-support prompt 组装
- 安全预检查路径
- 手机端持有 session 上下文处理
- 基础测试
- 本地设置 README

等这个后端接口稳定后，再开始微信小程序集成。
